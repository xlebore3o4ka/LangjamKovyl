# Архитектура runtime

Chompo использует двухуровневую модель имён и набор независимых hot-path оптимизаций, чтобы совместить быстрое исполнение с расширяемостью host API.

## Pipeline

```text
source
  -> Lexer
  -> Parser / AST
  -> Resolver
  -> optimized Interpreter
```

Resolver не исполняет код и не знает о конкретных встроенных функциях. Он один раз вычисляет лексический адрес каждого имени и количество slots для runtime scope.

## Локальные переменные

Локальная ссылка хранит пару:

```text
(depth, slot)
```

- `depth` — сколько реальных `Environment` пройти к родителю;
- `slot` — индекс значения в плотном массиве нужного `Environment`.

Во время исполнения локальное чтение не хеширует строку и не ищет имя по цепочке map. `Environment::get_ref()` возвращает прямую ссылку на значение, поэтому присваивания и `++/--` могут изменять slot без промежуточного callable target.

Модель поддерживает:

- параметры;
- вложенные блоки;
- shadowing;
- рекурсию;
- closures;
- функции, объявленные внутри блоков.

## Удаление пустых runtime scope

Resolver создаёт lexical scope для блока только тогда, когда блок непосредственно объявляет `var` или `fun`.

```javascript
while (condition) {
    value++;
}
```

Такой блок не создаёт новый `Environment` на каждой итерации. Одновременно Resolver не учитывает его в `depth`, поэтому адреса внешних переменных остаются корректными.

Блок с собственными объявлениями сохраняет обычную лексическую семантику и получает заранее вычисленное число slots.

## Кеширование литералов

Числа, строки и `char` декодируются при первом вычислении узла и сохраняются в `LiteralExpr`. Повторное выполнение тела цикла не запускает `from_chars` и не разбирает escape-последовательности заново.

## Пулы окружений

Runtime переиспользует завершённые:

- block environments;
- iteration environments;
- function frames.

Environment возвращается в pool только при `use_count() == 1`. Если его удерживает closure, frame не очищается и не выдаётся другому вызову.

## Control flow без исключений

`return`, `break` и `continue` являются нормальным управлением потоком и передаются через компактное состояние `ControlFlow`.

C++ exceptions остаются для настоящих ошибок:

- неверных типов;
- деления на ноль;
- ошибок файлов/сети;
- выхода за границы;
- stack overflow runtime.

Это особенно ускоряет функции с ранним `return` и циклы с частыми `continue`.

## Арифметические fast paths

Для двух `integer` runtime сразу выполняет специализированную операцию над `int64_t`, минуя общую цепочку динамических числовых проверок.

`++` и `--` изменяют `integer`, `double` и `char` напрямую. Общий operator dispatcher используется только для остальных числовых комбинаций.

## Вызовы функций

Для каждого уровня call depth переиспользуется собственный argument vector. Это убирает повторные heap allocations при большом количестве вызовов.

User function получает frame с точным числом slots, вычисленным Resolver. Завершённые незахваченные frames возвращаются в pool.

## Глобальные и native-имена

Глобальные значения остаются в расширяемом реестре по `SymbolId`.

```cpp
interpreter.install_collection_builtins();
interpreter.install_io_builtins(io_manager);
interpreter.install_network_builtins(network_manager);
```

Новый native-модуль по-прежнему регистрирует функции строковыми именами. Для добавления встроенной функции не требуется изменять Resolver, AST или формат local slots.

Каждый `Environment` хранит прямой указатель на global root, поэтому глобальный доступ не обходит цепочку родителей.

## Совместимый fallback

`Environment` сохраняет map-based dynamic path для host-кода, глобального registry и будущих динамических модулей. Код, прошедший Resolver, использует быстрые slots; неразмеченные host-обращения могут использовать прежний поиск.

Это оставляет путь к:

- модулям и import;
- классам и методам;
- plugin API;
- REPL с динамическими объявлениями;
- опциональному bytecode backend;

без возврата локальных переменных к строковым lookup.

## Семантика

Оптимизация сохраняет:

- порядок вычисления выражений;
- правила shadowing;
- ссылочную семантику массивов;
- snapshot-семантику `for-in`;
- поведение closures;
- runtime-ошибки неопределённых и повторно объявленных переменных;
- API встроенных функций.
